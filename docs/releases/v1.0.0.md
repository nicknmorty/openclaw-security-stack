# OpenClaw Security Stack v1.0.0

Working public v1 of the local-first OpenClaw security scanner stack.

This release gives operators one read-only command for reviewing an OpenClaw
repo or runtime-adjacent project, producing redacted local reports, tracking
finding state across runs, acknowledging accepted noise, and sharing a short
digest with trusted reviewers.

Scope note: v1 is best described as an Anthropic-inspired OpenClaw local
scanner stack plus staged OpenClaw workflow wrappers. It is not yet a full
implementation of Anthropic's autonomous Python/Docker/gVisor defending-code
reference harness.

## Highlights

- Adds the `node scripts/security-scan.mjs` front door.
- Runs four read-only lanes by default:
  - `threat-model`
  - `static-scan`
  - `supply-chain`
  - `runtime-health`
- Writes consolidated local output:
  - `runs/summary/DIGEST.txt`
  - `runs/summary/SUMMARY.md`
  - `runs/summary/SUMMARY.json`
  - per-lane `REPORT.md` and `REPORT.json` files
- Tracks findings as `new`, `persistent`, `resolved`, and `acknowledged`.
- Namespaces state per scanner lane so one tool does not resolve another
  tool's findings.
- Supports `security-suppressions.json` for exact fingerprint suppressions,
  match-rule suppressions, reasons, and optional expiry.
- Adds public-safe operator docs and examples:
  - `runbooks/v1-operator-runbook.md`
  - `docs/V1_ACCEPTANCE.md`
  - `security-scan.config.example.json`
  - `security-suppressions.example.json`
  - `reference/security/identity-denylist.example.json`
- Adds a root MIT license and third-party notices for vendored/adapted upstream
  material.

## Scanner Lanes

- `threat-model`: static project inventory and threat-surface notes.
- `static-scan`: deterministic code-pattern candidate findings.
- `supply-chain`: npm/Python manifest and lockfile posture checks.
- `runtime-health`: portable baseline checks for listening sockets, SSH config,
  SSH private-key permissions, opt-in sensitive-file permissions, and firewall
  presence.

All lanes are designed to be local and read-only in v1. The scanner does not
patch files, change host policy, edit cron, restart services, or call external
advisory APIs.

`runtime-health` is host posture, not target-repo code analysis. It runs by
default in v1, so use `--scanners threat-model,static-scan,supply-chain` when
you want a repo-only scan.

The mutating guard helpers, such as `security-secret-guard.sh --remediate` and
`oauth-containment-audit.sh --fix`, are separate operator remediation tools and
are not part of the normal scanner path.

## Quick Start

```bash
node scripts/security-scan.mjs --target /path/to/repo --label my-repo
```

Configured target flow:

```bash
cp security-scan.config.example.json security-scan.config.json
node scripts/security-scan.mjs
```

Suppress accepted noise:

```bash
cp security-suppressions.example.json security-suppressions.json
node scripts/security-scan.mjs --suppressions security-suppressions.json
```

## Known Limits

- Findings are review candidates, not proof of exploitability.
- Supply-chain checks are deterministic posture checks, not CVE/advisory
  intelligence.
- Runtime-health is a portable baseline, not a complete host hardening audit.
- Anthropic upstream drift is not automatically tracked; alignment claims need
  explicit vendored-source review.
- Scheduling/cron wiring is intentionally left to operators.
- v1 performs no automated remediation.

## Post-Tag Hardening On Main

After the `v1.0.0` tag, `main` received follow-up hardening that is documented
in `CHANGELOG.md`:

- Runtime-health socket fingerprints now include protocol, address, and port so
  per-port suppressions and state tracking do not collapse unrelated sockets.
- Persisted `STATE.json` now receives the same redaction pass as `REPORT.md`
  and `REPORT.json`.
- Runtime-health redaction tests no longer assert on the live username, avoiding
  false failures on root-runner hosts where SSH root-login findings legitimately
  contain the word "root".

Operators should use current `main` for the latest v1 hardening unless they
specifically need the original tagged snapshot.
